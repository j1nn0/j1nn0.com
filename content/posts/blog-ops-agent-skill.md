記事を出すたびに、AI エージェントに同じ指示をしていた。
タグは小文字アンダースコア区切りで3〜5個、front matter に draft を忘れない、文体はだ・である調、と。
指示を忘れると平気で破られて、直すのは自分だった。

このブログの運用を丸ごとエージェント用のスキル(blog-ops)に書き下したら、指示のし直しがなくなった。
やってみてわかったのは、AI に運用を任せる作業の本体がプロンプトの工夫ではなく、暗黙知の書き下しだということだ。
頭の中にしかなかった規約を「文章のルール」と「機械検査」に分解して外に出す。
それがほぼすべてだった。

できあがったスキルは [j1nn0/skills](https://github.com/j1nn0/skills/tree/main/skills/blog-ops) で公開している。

## 入口1枚、分冊5本、検査スクリプト1本

blog-ops は現在、次の構成になっている。
このディレクトリだけで合計631行ある。

```
blog-ops/
├── SKILL.md                  入口。段階ごとの振り分け表(74行)
├── references/
│   ├── new-post.md           新規記事の準備(スラッグ、front matter、summary、タグ、画像)(70行)
│   ├── publish-check.md      公開・コミット前のチェック(72行)
│   ├── social-announcement.md 記事公開後の SNS 告知文(27行)
│   ├── tags.md               タグの追加・改名・削除とリダイレクト(57行)
│   └── conventions.md       作業規約(スコープ、コーディング、コミット、セキュリティ)(56行)
└── scripts/
    └── check_posts.py        記事構造、本文リンク、画像、タグの機械検査(275行)
```

スキルというと Claude Code の機能に聞こえるが、blog-ops は特定のエージェントに依存していない。
自分は Claude Code のほかに Codex CLI や oh-my-pi、Antigravity、OpenCode も使っていて、どのエージェントで作業するかはその時々で変わる。
だから実体はエージェント中立の `.agents/skills/` に置き、各エージェントにはそこを読ませている。

入口の SKILL.md は、規約そのものをほとんど持っていない。
持っているのは「いまの作業がどの段階か」を判定して、読むべきファイルへ振り分ける表だけだ。

| 段階 | 読ませるもの |
|------|-------------|
| アイデアを企画に育てる | `blog-idea-grilling` スキルで企画カードを作る |
| 構成、タイトル、SEO、記事単位のレビュー | `blog-writing-guide-ja` スキル |
| 本文の文体、文レベルの推敲 | `writing-ja` スキル |
| 新規記事の準備 | `references/new-post.md` |
| 公開前チェック | `references/publish-check.md` |
| 公開後の SNS 告知 | `references/social-announcement.md` |
| タグの変更 | `references/tags.md` |
| 作業規約 | `references/conventions.md` |

更新で大きく変わったのは、入口に機能を足したことではない。
作業の判断単位に合わせて、企画、執筆、公開後の告知を別の読み物へ分けたことだ。

## 決まりきった検査を LLM にやらせない

front matter に title と date があるか。
タグが小文字アンダースコア区切りか。
スラッグが既存記事と重複していないか。
本文から参照した画像や記事が存在するか。

この手の検査は答えが一意に決まる。
一意に決まる検査を LLM にやらせる理由はない。
実行のたびに結果が揺れるし、全記事を読ませるトークンも毎回かかる。

そこで決定的に検査できる規約を `check_posts.py` に落とした。
現在のスクリプトは外部依存なしで、フラット記事とページバンドルのスラッグ、front matter、タグページ、本文の inline Markdown リンクまで検査する。
タグ数の下限と上限、タグの正規表現は CLI オプションで変更できる。

検査項目そのものより、ERROR と WARN の線引きに考えることが多かった。

| レベル | 例 | 意味 |
|--------|-----|------|
| ERROR | title や date の欠落、タグ表記違反、スラッグ重複、存在しない画像や内部リンク | 構造や参照が壊れている。修正する |
| WARN | タグ数、draft の欠落や過去日付、slug の欠落、孤児タグページ | 規約からの逸脱だが意図的な場合がある。人間が判断する |

この線を引いておくと、エージェントへの指示は「ERROR は修正必須、WARN は判断のうえ対応」の一行で済む。
逆に、文体や構成のような答えが一意に決まらない規約は、文章のままスキルに書く。
検査の分担を先に決めると、スキル本文には機械で検査できないものだけが残って薄くなる。

今回の更新では、文体にも機械検査を追加した。
`writing-ja/scripts/check_style.py` は内容のない予告、根拠のない強調、抽象的な宣伝語、240字を超える段落などを候補として WARN にする。
これは正誤を決める検査ではなく、文脈を見て人間が判断するための一覧だ。
決定的な記事構造の検査と、文体の見直し候補を同じスクリプトに詰め込まなかった。

## その規約、スキルに書くか AGENTS.md に書くか

一番迷ったのがこの判断だった。

「タグは小文字アンダースコア区切り」は Hugo ブログならどこでも通用する規約で、「リダイレクトは `static/_redirects` に書く」はこのサイトが Cloudflare Pages にあるから成り立つ事実だ。
両方を同じファイルに書くと、スキルをこのリポジトリの外へ持ち出せなくなる。

置き場所の基準はこう決めた。

| 置き場所 | 書くもの | 例 |
|---------|---------|-----|
| スキル(blog-ops) | Hugo ブログ一般に通用する既定 | タグ表記、front matter の形式、公開前チェックの手順 |
| リポジトリの AGENTS.md | そのサイト固有の事実と、既定からの逸脱 | Cloudflare Pages のリダイレクト方式、カスタム CSS の場所 |

受け皿を AGENTS.md にしたのは、特定エージェント専用の設定ファイルではなく、複数のエージェント CLI が共通で読むファイルだからだ。
衝突したら AGENTS.md が勝つ、という優先順位もスキル側に明記した。
サイト固有の事実はコードを見れば確かめられるが、スキルの汎用既定はどのリポジトリにも痕跡を残さない。
逸脱を書く場所は AGENTS.md 側にしか置けない。

この分離のおかげで blog-ops は j1nn0.com 専用ではなくなり、いまは汎用化した状態で j1nn0/skills リポジトリに置いている。

## SKILL.md に全部書かない

631行の規約と検査処理を1枚の SKILL.md にまとめると、タグを1個直すだけの作業でも全行がエージェントのコンテキストに載る。
コンテキストは有限で、いまの作業に関係ない規約が多いほど、関係ある指示が薄まる。

そこで入口には振り分け表だけを置き、具体的な手順は段階ごとの references に分冊した。
公開前チェックの72行は、公開前にしか読まれない。
SNS 告知の27行も、記事を公開したあとにしか読まれない。

企画の扱いも分けた。
以前は `grilling` スキルでアイデアを掘り下げていたが、現在はブログ記事向けの `blog-idea-grilling` を追加している。
このスキルはアイデアを採点するのではなく、読者の変化と一文の主張を企画の芯として整理する。
既存ファイルから確認できる事実は調べ、企画を変える選択だけを一問ずつ尋ねる。
最後に、支える材料、扱う範囲、要確認事項を含む企画カードを `blog-writing-guide-ja` へ渡す。

スキルが増えたときの衝突にも手を打った。
企画と構成は `blog-writing-guide-ja` が正、文レベルの事実関係と論理と文体は `writing-ja` が正、それ以外の文章系スキルは使わない、と `blog-ops` に書いた。
書いておかないと、どのスキルを読むかをエージェントが毎回選び直すことになる。

## この規約、書いたのは自分ではない

ここまでの判断を並べると自分で書き上げたように見えるが、blog-ops の文章と構造を書いたのは Claude Code(Fable 5)だ。
規約の文章化も、入口と分冊の構造化も、検査スクリプトの実装も任せた。

それで人間の仕事がなくなったかというと、逆だった。
文章化は任せられるが、境界を引く判断は最後まで自分に残った。
どの規約が汎用で、どれがこのサイト固有か。
スキル同士が重なったとき、どちらを正とするか。
決定的な検査と、人間が読むべき候補をどこで分けるか。
つまり前の3節に書いた内容こそが、AI に渡せなかった部分だ。

考えてみれば当然で、暗黙知の境界線は自分の頭の中にしかなく、AI はそれを推測するしかない。
推測で置かれた境界は、それらしく見えても自分の運用と合わない。
書き下しを任せるときの人間の仕事は、執筆ではなく境界の判定に集中する。

## 効果はまだ1記事分。それでも消えたもの

正直に書くと、blog-ops を通して出した記事はこれが最初で、効果を数字で語れる段階にない。

それでも設計の時点で消えたものはある。
毎回の指示のし直しはなくなった。
規約はスキルに書いてあり、エージェントが自分で読むからだ。
記事構造やリンクのような決定的な規約は `check_posts.py` が公開前に止める。
文体の候補は `check_style.py` が拾い、人間が文脈を見て判断する。

初稿の作成時点では、企画は `grilling` スキルの質問攻めで固め、構成とタイトルは `blog-writing-guide-ja`、本文の文体は `writing-ja` に従っていた。
現在の流れでは、企画の芯を `blog-idea-grilling` の企画カードにまとめてから構成へ渡す。
スキルの更新で変わったのは、質問を増やしたことではなく、企画を次の工程へ渡す形式を決めたことだ。

運用が記事数本分たまったら、続きを書くつもりだ。

## 真似するなら検査スクリプトから

全部を一度に整える必要はない。
効きが分かりやすいのは機械検査で、自分のリポジトリの「決定的に検査できる規約」を1本のスクリプトに書き出すだけでも、エージェントへの指示は「これを通せ」に縮む。

ただし、検査スクリプトが返す WARN まで自動修正の命令にしてはいけない。
タグ数や draft の扱いには意図があり、文章の候補には文脈がある。
ERROR と WARN の意味を分け、修正する責任を人間側に残す。

次に、規約を汎用と固有に分ける。
Hugo ブログ全般の既定はスキルに置き、ホスティングやサイト固有の例外はリポジトリの AGENTS.md に置く。
タグ数やタグ表記がサイトに合わない場合は、`check_posts.py` の CLI オプションで検査条件を合わせる。

最後に、入口と分冊に分ける。
企画カード、本文の文体、公開前チェック、SNS 告知のように読むタイミングが異なるものを同じ SKILL.md に詰め込まない。
エージェントがいま必要な規約だけを読めるようにする。

blog-ops の実物(SKILL.md、references、検査スクリプト)は [j1nn0/skills](https://github.com/j1nn0/skills/tree/main/skills/blog-ops) に置いてある。
規約の中身はあなたのブログでは違うはずだが、文章の規約と機械検査に分ける、汎用と固有を分ける、入口と分冊に分ける、という3つの分け方はブログ以外の繰り返し作業にも持っていける。

少なくとも自分は、もう同じ指示を打ち直していない。
